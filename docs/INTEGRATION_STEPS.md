# Email to TSP Optimizer - Integration Steps

This document describes how to integrate a Python tool that:

1. Receives an email with date, departure time, and stop addresses.
2. Uses `gpt-4o-mini` to extract structured data.
3. Saves extracted data to CSV.
4. Calls openrouteservice APIs to optimize the route (TSP).
5. Replies to the original sender with optimal stop order and useful route details.

## 1) Functional Scope (MVP)

**Input**
- Raw email content (subject + plain text body or HTML converted to plain text).
- Example format includes Spanish labels (`Fecha`, `Hora`) and bullet-list addresses.

**Output**
- Structured trip data (date, departure time, addresses).
- CSV record(s) for traceability/history.
- Optimized stop order from openrouteservice.
- Reply email with:
  - ordered stops
  - estimated total duration/distance
  - optional map links and notes

---

## 2) High-Level Architecture

1. **Email Ingestion Layer**
   - Option A: Poll inbox via IMAP (simple MVP).
   - Option B: Provider webhook (Gmail Pub/Sub, Microsoft Graph subscriptions, etc.) for near-real-time.
2. **LLM Extraction Layer (`gpt-4o-mini`)**
   - Parse natural language email into strict JSON schema.
3. **Data Persistence Layer**
   - Save extraction + metadata to CSV.
4. **Routing Layer (openrouteservice)**
   - Geocode addresses (if needed) and run optimization endpoint.
5. **Notification Layer**
   - Send response email to original sender (same thread).
6. **Observability Layer**
   - Structured logs + status fields for each message.

## 4) Data Contract (Extraction Schema)

Use one strict schema as the contract between LLM and app:

- `request_id` (string, generated by app)
- `email_subject` (string)
- `email_from` (string)
- `trip_date` (string, `YYYY-MM-DD`)
- `departure_time` (string, `HH:MM` 24h local time)
- `timezone` (string, e.g. `America/Santiago`)
- `addresses` (array of strings, min 2)
- `language_detected` (string, e.g. `es`)
- `confidence` (number 0-1)
- `warnings` (array of strings)

Validation rules:
- Date/time must be normalized to ISO-compatible fields.
- At least 2 valid addresses required to optimize.
- If ambiguous date (no year), infer current year using `APP_TIMEZONE`, and flag in `warnings`.

## 5) GPT-4o-mini Extraction Strategy

1. Build a robust prompt with:
   - role: extraction engine
   - hard requirement: return **valid JSON only**
   - expected schema with field-level constraints
   - language instruction: preserve original addresses exactly
2. Provide email subject + cleaned body.
3. Reject non-JSON outputs and retry with a stricter repair prompt.
4. Validate JSON using schema validation before downstream usage.
5. If validation fails after retries:
   - mark request as `FAILED_EXTRACTION`
   - send polite clarification email asking for corrected format.

---

## 6) CSV Persistence Design

Two CSV outputs are recommended:

1. **Requests CSV** (`requests.csv`) - one row per email
   - request metadata, date/time, sender, extraction confidence, status, total stops
2. **Stops CSV** (`stops.csv`) - one row per stop
   - request_id, original_index, address, lat, lon, optimized_index

Benefits:
- easy auditing and reprocessing
- simple BI/reporting later
- supports debugging extraction/routing quality

## 7) openrouteservice Integration Flow

openrouteservice provides services for geocoding, matrices, and optimization (VRP/TSP-oriented workflows). See official site and docs:
- [openrouteservice](https://openrouteservice.org/)

### Routing sequence
1. **Geocode each address** to coordinates.
   - Use deterministic matching policy.
   - If multiple results, choose best score and keep confidence note.
2. **Build optimization payload**
   - Single vehicle for TSP-like route.
   - Define start time based on extracted date/time.
   - Include all stops as jobs.
3. **Call optimization endpoint**
   - Get optimized sequence and route summary.
4. **Map results back to original addresses**
   - produce ordered stop list with indices and coordinates.
5. **Optional enrichment**
   - total distance/time in human-readable format
   - per-leg ETA windows
   - shareable map links (if available in your flow)

## 8) Email Reply Content (Template Guidance)

Reply in the same email thread with:

1. **Header**
   - Trip date/time and timezone interpreted by the system
2. **Optimal order**
   - numbered list of stops (1..N)
3. **Summary**
   - total distance
   - estimated total travel time
   - number of stops
4. **Data quality notes**
   - any addresses with low geocoding confidence
   - assumptions (e.g., year inferred)
5. **Optional next step**
   - ask sender to confirm if first stop should be fixed as departure depot

## 9) Error Handling and Fallbacks

Define clear statuses:
- `RECEIVED`
- `EXTRACTED`
- `EXTRACTION_REVIEW_NEEDED`
- `GEOCODING_FAILED`
- `OPTIMIZED`
- `REPLIED`
- `FAILED`

Fallback behavior:
- If 1-2 addresses fail geocoding, continue with valid ones only if above minimum threshold and mention exclusions.
- If optimization API is unavailable, return partial response with extracted stops and "optimization pending".
- Always log raw error payloads (without leaking secrets).

## 10) Security and Compliance Basics

- Store API keys in environment variables/secrets manager, never in repo.
- Redact personal data in logs where possible.
- Validate and sanitize inbound email body before processing.
- Add rate limits and retry with exponential backoff for external API calls.

## 14) Example of Expected Structured Extraction

For the sample email, normalized output should look conceptually like:

- `trip_date`: `2026-03-19` (example year inference)
- `departure_time`: `10:00`
- `timezone`: `America/Santiago`
- `addresses`: 10 extracted strings in original order

The optimization step then computes the best visitation order for those 10 stops.
